## 0. 이 업무가 필요한 이유 (Why)

엠플랜잇 GA 랜딩플랫폼은 "**섹션을 메뉴판처럼 선택해 조립하는 랜딩 빌더**" 를 지향한다. 즉, 하나의 `design_v1.0` 을 쓰는 게 아니라 **리드폼 3~5종, 히어로 3~5종, 협력사월 3~5종 …** 을 갖춘 라이브러리에서 클라이언트가 골라 쓰는 구조로 진화한다.

이를 위해 먼저 **시장에서 실증된 섹션 패턴**을 섹션별로 3~5개씩 수집·분류·태깅해 Figma 보드 1장으로 정리한다. 본 리서치의 결과물이 곧 **랜딩플랫폼 v2 컴포넌트 라이브러리의 입력**이 된다.

본 브리프는 "무엇을 / 어디서 / 어떻게 / 얼마나" 에 대한 명시적 지시서다. 애매한 표현은 두지 않는다. 막히는 부분은 즉시 PM 에 문의한다.

## 3. 섹션 라이브러리 — 16종 리서치 가이드 (How)

각 섹션마다: **찾는 것 / 변형 축 / 수집 수 / 필수 태그**를 명시. 변형 축(Variant Axis)을 기준으로 서로 다른 패턴을 의도적으로 선택한다 ("다 똑같은 오른쪽 사이드폼" 3개는 리서치 실패).

### §1. GNB / 네비게이션
- 찾는 것: 로고·메뉴 구성·CTA 버튼 위치·모바일 햄버거 처리
- **변형 축**: (a) 카테고리 드롭다운 유/무 (b) 고정 sticky + 투명 vs 솔리드 (c) 검색창 포함 여부
- 수집: **3~4개**
- 필수 태그: `menu_depth=1|2|3`, `sticky=y/n`, `cta_count=0|1|2`, `logo_position=left|center`

### §2. Hero — 메인 카피 + 시각 요소
- 찾는 것: 타이틀 길이·보조 카피·이미지/일러스트/영상·CTA 버튼 수
- **변형 축**: (a) Split(좌우) vs Full-bleed (b) 이미지 실사 vs 일러스트 vs Lottie (c) 배경 그라데이션 vs 단색 vs 비디오
- 수집: **4~5개**
- 필수 태그: `layout=split|full|stacked`, `visual=photo|illust|lottie|none`, `cta_count`, `headline_lines=1|2|3`

### §3. 리드폼 — Hero 인접형 ★★★
- 찾는 것: 필드 수·필드 순서·검증·동의 UX·CTA 문구
- **변형 축**: (a) **오른쪽 sticky 사이드** (캐롯·키움) (b) **히어로-as-폼 split** (DB) (c) **인라인 하단 단순 폼** (일부 GA) (d) **2단 wizard** (핀테크)
- 수집: **5개** (각 variant 1개 이상)
- 필수 태그: `position=right_sticky|inline|as_hero|modal|wizard`, `field_count=3|4|5|6+`, `consent_ui=inline|accordion|modal`, `cta_label`

### §4. 리드폼 — 본문 중단형
- 찾는 것: 2차 리드폼(다크 배경) 패턴
- **변형 축**: 다크 vs 그라데이션 vs 인버스 / 긴 폼 vs 미니멀
- 수집: **3개**
- 필수 태그: `tone=dark|brand|light`, `field_count`, `cta_label`

### §5. 왜 비교·수치 카드 (Why Us / Trust Metrics)
- 찾는 것: 큰 숫자 + 라벨 + 기준일자 카드 3~4개
- **변형 축**: (a) 2×2 그리드 vs 가로 4열 (b) 아이콘 유무 (c) 배경 틴트 유무
- 수집: **3~4개**
- 필수 태그: `count=2|3|4|5`, `icon=y/n`, `bg=white|tint|dark`

### §6. 상품 안내 카드 그리드
- 찾는 것: 상품 카테고리 카드 (아이콘·이름·1줄 설명·CTA)
- **변형 축**: (a) 4열 vs 3열 vs 2열 (b) 카드 크기 vs 높이 균일 (c) 카드 내 링크 1개 vs 복수 (d) 아이콘 스타일 (line / filled / 일러스트)
- 수집: **3~4개**
- 필수 태그: `columns=2|3|4`, `icon_style=line|filled|illust|photo`, `card_height=fixed|flexible`

### §7. 협력사 로고월
- 찾는 것: 제휴 브랜드 배치 방식
- **변형 축**: (a) 생명·손해 2컬럼 분리 (메타리치) (b) 단일 그리드 (c) 컨베이어 애니메이션 띠 (d) 그레이스케일 vs 컬러
- 수집: **3~4개**
- 필수 태그: `layout=grid|strip|separated`, `filter=grayscale|color`, `count=15|30|50+`

### §8. 절차 프로세스 (Step Flow)
- 찾는 것: 상담 → 설계 → 가입의 N단계 표현
- **변형 축**: (a) 3단계 vs 5단계 (b) 가로 타임라인 vs 세로 (c) 원형 번호 vs 카드 (d) 아이콘 포함 vs 텍스트만
- 수집: **3~4개**
- 필수 태그: `steps=3|4|5|6`, `orient=horizontal|vertical`, `marker=circle|square|icon`

### §9. 상담사·전문가 소개
- 찾는 것: 사람 사진 + 이름 + 경력 카드
- **변형 축**: (a) 슬라이더 vs 그리드 (b) 사진 스타일 (정장·배경 제거·일러스트) (c) 카드 내 경력 bullet 수
- 수집: **2~3개** (메타리치 외 확보 어려울 수 있음)
- 필수 태그: `style=carousel|grid`, `photo_type=photo|illust`, `bullet_count`

### §10. 데이터 교육·통계 섹션
- 찾는 것: 차트·통계·"왜 지금 필요한가" 설득 콘텐츠
- **변형 축**: (a) 정적 SVG 차트 (b) 인터랙티브 차트 (c) 다이어그램·일러스트 (d) 다크 배경 vs 라이트
- 수집: **3개**
- 필수 태그: `chart_type=bar|line|pie|none`, `interactive=y/n`, `bg=dark|light`

### §11. 상품 비교 카드 (3~5 보험사)
- 찾는 것: 보험사별 보장·보험료 비교
- **변형 축**: (a) 가격 공개 vs 마스킹 (***,***원) vs "상담 안내" (b) 카드 3개 vs 5개 (c) 특약 bullet 수
- 수집: **3개**
- 필수 태그: `price_reveal=open|masked|hidden`, `card_count`, `feature_count`

### §12. 추천 대상 페르소나 카드
- 찾는 것: "이런 분들께 추천" 카드
- **변형 축**: (a) 일러스트 vs 사진 vs 아이콘 (b) 카드 3개 vs 4개 vs 그 이상 (c) 상황 기반 vs 연령 기반 설명
- 수집: **2~3개**
- 필수 태그: `count=3|4|5`, `visual=illust|photo|icon`, `axis=age|situation|job`

### §13. FAQ (아코디언)
- 찾는 것: 질문 5개 안팎, 아코디언 펼침 UX
- **변형 축**: (a) 숫자 번호 유무 (b) 질문 분류 태그 유무 (c) 카드형 vs 라인 디바이더형
- 수집: **3개**
- 필수 태그: `style=card|line`, `categorized=y/n`, `count=4|5|6|10+`

### §14. 후기·Testimonial
- 찾는 것: 사용자 후기 카드
- **변형 축**: (a) 실명 공개 vs 마스킹 (b) 평점 별표 vs 감정 아이콘 vs 없음 (c) 날짜 노출 유무
- 수집: **2~3개** (삼성화재 6개 실명+날짜는 이미 03 문서에 있음)
- 필수 태그: `identity=real|masked`, `rating=y/n`, `date=y/n`

### §15. 모바일 하단 sticky CTA 바
- 찾는 것: 모바일 뷰 고정 CTA 바
- **변형 축**: (a) 단일 버튼 vs 듀얼 버튼 (b) 투명·반투명·솔리드 (c) 전화 아이콘 포함 여부

### §16. Footer — 준법 고지 + 사업자 정보
- 찾는 것: 준법 문구 배치·사업자 정보 구조·링크 계층
- **변형 축**: (a) Variant A/B/C (기획 04_법적고지 참조) (b) 컬러 다크 vs 라이트 (c) 사업자정보 1블록 vs 여러 블록

## 4. 캡처 메타데이터 규칙 (How — 태깅)

모든 레퍼런스 프레임 아래에 **메타데이터 박스**를 고정 배치. 포맷:

```
┌─────────────────────────────────────────────────┐
│ [브랜드]   삼성화재 다이렉트                     │
│ [URL]      direct.samsungfire.com (2026.04 캡처)│
│ [섹션]     §3 리드폼 — Hero 인접형               │
│ [variant]  as_hero (폼-as-hero split)           │
│ [tags]     position=as_hero, fields=4, consent=inline│
│ [특이점]   생년월일을 탭 전환으로 처리            │
│ [평가 Y/N] 적합◯ 독창◯ 완성도◯ 재현◯ (4/4)      │
│ [강도]     ★★★★★  (루브릭 §4.1 기준)            │
│ [이유]     GA 전환 흐름 일치 + 폼-as-hero 신선 + 완성도 高 + 스택 재현 가능 │
│ [채택안]   Yes / No / 조건부                     │
└─────────────────────────────────────────────────┘
```

- **채택안**: Yes = v2 라이브러리 후보 / No = 제외 / 조건부 = PM 협의

### 4.1 강도(★) 평가 루브릭 — 명시 기준

> 주관 판단 금지. 아래 4개 문항에 **YES/NO** 로 답하고, YES 개수로 **정확히 한 등급**을 산출한다. 애매하면 낮은 등급으로.

**판별 4문항** (각각 Y/N 체크):
1. **GA 적합도** — 보험 GA·중개·상담 연결 목적·신뢰·전문형 톤·전환 흐름과 일치하는가?
2. **변형 축 독창성** — 이미 보드에 들어간 다른 variant 와 확실히 다른 패턴인가? (같은 variant axis 로 이미 1건 있으면 NO)
3. **UX·비주얼 완성도** — 반응형 안정·접근성 기본·정보 위계 명확·오타/깨짐/괴리 없음?
4. **재현 가능성** — 우리 스택(정적 HTML + CSS Variables + Pretendard + Lucide + Formspree)으로 합리적 비용 내 구현 가능한가?

**등급 산출표**:

| 등급 | Y 개수 | 정의·조치 |
|---|---|---|
| **★★★★★** | 4/4 | 즉시 v2 라이브러리 채택 권고. 그대로 variant 템플릿 후보. |
| **★★★★** | 3/4 | **조건부 채택**. 부족 1곳이 수정·대체 가능한 경우. [이유] 에 수정 포인트 명기. |
| **★★★** | 2/4 | 참고·학습 수준. v2 후반·v3 재검토. 지금은 채택 보류. |
| **★★** | 1/4 | 단편 요소(컬러·타이포·미세 UX)만 참고. 프레임 자체는 비채택. |
| **★** | 0/4 **또는** 다크패턴·허위광고·저작권 소지 | **기록 보존용 제외 대상**. `reject_reason` 필수 기재. |

**추가 규칙**:
- Y/N 는 **근거 한 줄**을 메타박스 `[이유]` 에 기재. "왠지 좋아 보임" 류 금지.
- **디자이너 1차 채점 → PM 2차 채점** (리뷰 시). 2등급 이상 차이 나면 재논의·합의 등급 기록.
- 동일 브랜드의 같은 variant 가 이미 있으면 2번째는 자동 **★★ 이하**. 중복 방지.
- ★★★★★ 가 전체의 30% 를 넘으면 평가 기준이 느슨한 것. PM이 재검수 요청.
- ★ 만 붙은 항목도 **제외 이유(다크패턴 종류, 법적 리스크 등)** 를 반드시 기재.

**악용 방지**: 디자이너 본인이 좋아하는 스타일에 ★ 를 몰아주는 것이 가장 흔한 실패. 4문항 체크로 우회 방지.

### 4.2 캡처 규칙
- **전체 페이지 캡처**: 데스크톱 1440px 폭 + 모바일 375px 폭 각 1장
- **섹션 단독 컷**: 해당 섹션만 crop 한 고해상도 1장
- 필요 시 **인터랙션 시퀀스**: hover / focus / 제출 / 모달 3~5프레임
- **개인정보 마스킹**: 후기 실명·전화번호 등 일부 마스킹 필수 (본인 캡처 이미지라도)

### 4.3 컬러·타이포 샘플링 (Page 2)
- 브랜드별 Primary / Secondary / Accent 컬러 스포이트 (Figma 내장)
- 사용 폰트 추정 (Pretendard / 나눔 / Spoqa / 커스텀)
- 헤드라인 크기 / 본문 크기 간단 메모

## 7. 금지 사항 (Hard NOs)

- **카피캣 금지**: 픽셀 단위 복제·레이아웃 그대로 베끼기는 저작권·상표권 위반 소지. "구조·패턴 참고" 만 허용.
- **실제 브랜드 로고 사용 금지**: 보드에는 스크린샷으로만 존재. 우리 v1.0 구현에 실 로고 이미지 이식 금지.
- **다크패턴 참고 제외**: "X 버튼 숨김", "취소 버튼 색상 약화" 등 UX 오용 사례는 **`다크패턴` 태그로 별도 표시하고 제외 권고**.
- **저작권 이미지 저장 금지**: 고객 후기·상담사 사진은 캡처 시 워터마킹 또는 모자이크.
- **확인되지 않은 주장 포함 금지**: "업계 1위"·"만족도 N%" 같은 수치는 출처 없는 경우 **붉은 태그**.
