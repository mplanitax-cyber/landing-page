## 0. 한 장 요약 (TL;DR)

브리프 §1 산출물 "Figma 보드 1장 × 약 60개 레퍼런스 캡처"를 Cowork(Claude)로 **완전 자동화**하려고 시도했을 때 현 시점에 만난 두 가지 블로커가 있다.

1. **프로젝트 지정 Figma 파일에 쓰기 권한 없음** — 링크 `RuIrn6AadbfyP4VAfVWDZM/레퍼런스`는 `hyejin.kim@mplanit.co.kr` 계정으로 접근 불가 (Figma MCP 인증 주체). 해당 파일이 다른 팀/조직 소유이거나 편집자 초대가 없는 상태.
2. **웹 자동 캡처 채널 미연결** — Claude in Chrome 확장이 현재 로컬 Chrome에 연결되어 있지 않아, 60+ 웹사이트 자동 방문 캡처 파이프라인을 지금 이 턴에서 실행할 수 없음.

그럼에도 불구하고 **즉시 실행 가능한 경로 3가지**가 있고, 그중 하나인 **반자동 하이브리드**가 품질·속도·통제권 측면에서 브리프 마감(4/28) 안에 가장 안정적이다. 본 문서는 그 판단 근거와 함께, 브리프 §1.1의 Figma 페이지 구조를 **스켈레톤 단계까지 그대로 옮길 수 있는 가이드**를 제공한다.

### 1.1 경로 A — 완전 자동 (현재 불가)
Cowork가 단독으로 브리프 §1~6을 처음부터 끝까지 수행하는 시나리오.

필요 조건:

- Chrome 확장이 로컬 Chrome에 **연결**되어 있을 것 (메뉴 → Cowork → Browser에서 활성화).
- 대상 18개 브랜드 사이트가 **봇 차단 없이 접근 가능**할 것 (일부 공공·로그인 필요 사이트 예외 주의).

현 블록 상태:

- Chrome MCP `tabs_context_mcp` 호출 결과 `Claude in Chrome is not connected`.

### 1.2 경로 B — 반자동 하이브리드 (추천)
Cowork가 **구조·태깅·큐레이션 뼈대**를 자동 생성하고, **캡처만 디자이너가 단시간에 수동**으로 붙이는 방식. 품질 통제권을 디자이너가 쥔 채 Cowork는 반복 작업(메타박스 복제, 태그 스키마, 큐레이션 표 채우기)을 제거한다.

Cowork가 자동 수행:
1. 신규 Figma 파일 생성 (또는 초대받은 파일에 페이지 추가)
2. Page 1~5 레이아웃 + 섹션 16종 × 프레임 스켈레톤 (1440×900 / 375×812 페어)
3. 각 프레임 하단 **메타데이터 박스 템플릿** 복제 배치 (§4 포맷 그대로 텍스트 프리필)
4. 태깅 컴포넌트/바리언트 세트 — `position=right_sticky|as_hero|inline|modal|wizard` 같은 Figma Variant Property 사전 선언
5. Page 5 큐레이션 노트 템플릿 16개 섹션 분량
6. 브리프 §6 품질 체크리스트 16 항목을 보드 내부에 체크박스로 심기
7. 대상 브랜드 18개 × 섹션 16종의 **캡처 큐 스프레드시트**(.xlsx) — 디자이너가 체크하며 진행

디자이너가 수동 수행:
- 브랜드 사이트 접속 → 섹션 단독 스크린샷 (데스크톱 + 모바일)
- Figma 프레임에 드래그-드롭
- 메타박스 4문항(Y/N) 체크 + `[이유]` 한 줄 기재

소요 추정: Cowork 자동 부분 30~60분, 디자이너 캡처·평가 1.5일 (섹션당 평균 3.75 프레임 × 약 2분).

### 1.3 경로 C — 수동 (브리프 원안 그대로)
브리프 §8 마일스톤 그대로. Cowork는 중간 체크포인트에서 **중복 variant 탐지, 루브릭 4문항 재채점, 메타박스 누락 스캔** 등 QA 역할만 수행. 마감 리스크가 가장 크다.

### 1.4 권고
**경로 B로 착수**를 권한다. Cowork가 뼈대를 만드는 30~60분 후 디자이너가 바로 캡처 작업에 들어갈 수 있어 D1(4/25)에 A군 8개사 마감 가능성이 가장 높다.

## 3. Figma 보드 구조 가이드 — v1 스켈레톤 설계서

아래는 경로 B 실행 시 Cowork가 그대로 생성하려는 Figma 보드의 스펙이다. 브리프 §1.1 구조 + §4 메타박스 + §4.1 루브릭을 **Figma 네이티브 컴포넌트**로 번역한 형태.

### 3.1 파일·페이지 구조

```
📄 엠플랜잇_랜딩_레퍼런스보드_v1
├─ Page 0: 사용 규칙·범례 (추가 권고) ← 등급 루브릭, 태그 사전, 체크리스트
├─ Page 1: 섹션 라이브러리         ← 본체. 16 섹션 × 3~5 variant
├─ Page 2: 브랜드 컬러·폰트 샘플링   ← 브랜드별 Primary/Secondary/Accent + 폰트
├─ Page 3: 인터랙션 캡처            ← hover/focus/submit/modal 시퀀스
├─ Page 4: 반응형 캡처              ← Desktop(1440) + Mobile(375) 쌍
└─ Page 5: 시니어 큐레이션 노트      ← 섹션 16개 권장·제외·주의 메모
```

> **추가 권고 — Page 0**: 브리프엔 없지만 "보드에 처음 들어오는 사람이 5분 안에 사용법을 이해할 수 있는 범례 페이지"를 앞에 두면 PM 리뷰·2등급 편차 논쟁을 확연히 줄인다. 루브릭 §4.1, 태그 스키마, 채택안 분류(Yes/No/조건부) 범례를 여기에 모은다.

### 3.2 Page 1 — 섹션 라이브러리 레이아웃

**16 섹션을 세로로 나열, 각 섹션은 제목 밴드 + 오른쪽으로 variant를 가로 배치**하는 큰 그리드 방식.

```
┌─────────────────────────────────────────────────────────────────┐
│ §1. GNB / 네비게이션                              [수집목표 3~4] │
│ ┌──Variant A──┐ ┌──Variant B──┐ ┌──Variant C──┐ ┌──Variant D──┐ │
│ │ 1440×900    │ │             │ │             │ │             │ │
│ │ [desktop]   │ │             │ │             │ │             │ │
│ │─────────────│ │             │ │             │ │             │ │
│ │ 375×812     │ │             │ │             │ │             │ │
│ │ [mobile]    │ │             │ │             │ │             │ │
│ │─────────────│ │             │ │             │ │             │ │
│ │ [메타박스]   │ │             │ │             │ │             │ │
│ └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘ │
├─────────────────────────────────────────────────────────────────┤
│ §2. Hero                                         [수집목표 4~5] │
│ ...                                                             │
└─────────────────────────────────────────────────────────────────┘
```

**프레임 치수 규칙** (브리프 §1.3):
- Desktop: `1440 × 900` 고정 프레임. 웹페이지가 세로로 더 길면 섹션 단독 crop으로 대체.
- Mobile: `375 × 812` 고정 프레임. 섹션 단독 crop.
- 메타박스: `320 × 220` (Auto Layout Vertical, 8px gap).
- Variant 간 가로 간격 48px, 섹션 간 세로 간격 120px.

### 3.3 섹션 제목 밴드 포맷

각 섹션 상단에 고정 밴드:

```
┌──────────────────────────────────────────────────────────────┐
│  §3. 리드폼 — Hero 인접형        ★★★  [수집 5개 / 현재 0/5]  │
│  변형축: position={right_sticky|as_hero|inline|modal|wizard} │
└──────────────────────────────────────────────────────────────┘
```

진척 카운터(`0/5`)는 디자이너가 프레임 채울 때 수동 업데이트하지 말고, Cowork에 "Page 1 variant 개수 집계 돌려" 하면 자동 갱신하는 스크립트로 해결하자. (`use_figma`로 실행)

### 3.4 메타데이터 박스 — Figma Component 형태

브리프 §4 박스를 **재사용 가능한 컴포넌트(Component)**로 등록. 모든 variant 프레임에서 인스턴스만 떨어뜨리면 된다.

**컴포넌트 이름**: `ref/meta-card`
**속성(Component Properties)**:
- `brand` (Text): 브랜드명
- `url` (Text): URL + 캡처일자
- `section` (Text): §번호 + 섹션 이름
- `variant` (Variant Property): 섹션별 enum 값 (`right_sticky`, `as_hero`, `inline`, `modal`, `wizard`, …)
- `tags` (Text): `position=..., fields=..., consent=...`처럼 key=value 콤마
- `note` (Text): 특이점 한 줄
- `y_ga` `y_origin` `y_polish` `y_feasible` (Boolean): 루브릭 4문항 Y/N
- `stars` (Variant Property): `★ | ★★ | ★★★ | ★★★★ | ★★★★★` (y_* 개수에서 자동 계산하도록 힌트 표시)
- `reason` (Text): [이유] 한 줄 (4문항 근거 종합)
- `adopted` (Variant Property): `Yes | No | 조건부`

**시각 레이아웃** (메타박스 내부):

```
┌────────────────────────────────────────────────────────┐
│ ▦ 삼성화재 다이렉트                    direct.samsung… │
│ §3 리드폼 — Hero 인접형 · variant=as_hero              │
│ tags: position=as_hero, fields=4, consent=inline       │
│ note: 생년월일을 탭 전환으로 처리                      │
│ ─────────────────────────────────────────────────────  │
│ [✓] GA 적합  [✓] 독창  [✓] 완성도  [✓] 재현            │
│ ★★★★★   ·   채택: Yes                                  │
│ reason: GA 전환 흐름 일치 + 폼-as-hero 신선 + 완성도…  │
└────────────────────────────────────────────────────────┘
```

**색상 규칙**: 채택=Yes는 녹색 좌측 바, 조건부는 앰버 바, No는 붉은 바. `stars=★` 이하면 카드 배경 그레이-틴트.

### 3.6 Page 5 — 큐레이션 노트 템플릿

브리프 §5의 마크다운 포맷을 **16개 섹션 모두에 반복 배치**. Figma 내부에서는 텍스트 레이어가 아니라 **각 섹션 영역별 Sticky Note 색상 구분** (Figma의 Section 기능 활용):

```
Section §3. 리드폼 — Hero 인접형
  🟢 채택 권고:  as_hero (DB) + right_sticky (캐롯)
  🔴 제외:       Wizard 3단계 이상
  🟡 주의:       가격 마스킹(***,***원)은 법무 승인 전까지 보류
  📝 이유:       GA Hero 인접 노출이 전환 유리, 2패턴 다 플랫폼에 내장
```

섹션 16개 전부를 위 포맷으로 복제. Cowork가 빈 템플릿 16개를 자동 배치하고, 디자이너는 내용만 채운다.

### 3.7 브리프 §6 품질 체크리스트 — 보드 내부화

16개 체크리스트를 Page 0 범례 끝에 **실시간 대시보드** 형태로 배치:

```
┌───── 제출 전 품질 체크 (16/16 모두 ✅ 면 OK) ─────┐
│ [✅] 1. 섹션 16종 전부 커버                        │
│ [▢] 2. 섹션당 variant 3개 이상                     │
│ [▢] 3. 브랜드 15개 이상                            │
│ [▢] 4. A군 GA 5개 이상                             │
│ ...                                                │
│ [▢] 16. 파일명 규칙 준수                           │
└────────────────────────────────────────────────────┘
```

Cowork가 Plugin API로 Page 1의 프레임 수·variant 수·브랜드 수를 집계해 체크박스를 자동 전환하는 스크립트를 제공한다. 디자이너가 수동으로 켜고 끌 필요 없음.
